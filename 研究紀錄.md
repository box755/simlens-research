## 2026-04-26 — 環境策略與第一步規劃

### 你的現況 vs 研究計畫的 Gap

| 元件 | 目前 | 研究計畫要求 | Gap |
|---|---|---|---|
| Whisper | 調 API | 本地 Whisper-Large-v3(含時間戳) | 換成本地 |
| LLaVA-NeXT | Replicate API | 本地 LLaVA-NeXT-13B(每 10 秒分段) | 換成本地 |
| LLM (生評論) | Gemini API | **本地 Llama-3.2-3B + 8 LoRA**(要訓練) | 從零訓練 |
| Reward judge | 無 | 本地 Qwen3-32B-Q4 | 從零部署 |
| Teacher | 無 | Claude API 蒸餾資料生成 | 必要的 API(一次性 ~$115) |
| GPT-4o spot-check | 無 | 200 樣本(~$5) | 必要的 API |

**結論**：3090 24GB 是這個研究計畫剛好可行的硬體底線。Whisper、LLaVA-NeXT、Llama-3.2-3B+LoRA、Qwen3-32B-Q4 都能塞進去（不同時跑），唯獨 Claude 蒸餾跟 GPT-4o spot-check 是**研究方法本身要求**的 API call，省不掉但都是一次性 + 小成本。

### VRAM 預算盤點(3090 24GB)

| 模型 | VRAM 估計 | 何時用 |
|---|---|---|
| Whisper-Large-v3(FP16) | ~10GB | Stage A 一次跑完 100 影片轉錄 |
| LLaVA-NeXT-13B(4-bit) | ~10GB | Stage A 跑完段描述後就可釋放 |
| Llama-3.2-3B(4-bit GPTQ) + LoRA | ~6.5GB | Stage B 訓練 + 推理 |
| Qwen3-32B(Q4_K_M, GGUF/Ollama) | ~20GB | Phase 2 reward judge(**單獨跑**) |

**關鍵設計**：這四個模型**不可同時駐留在 VRAM**。要按 pipeline 階段切換 — 跑 Whisper 時別跑 LLaVA、跑 Qwen judge 時別跑 Llama。研究計畫已經是這個邏輯（按 Week 編排），所以剛好對得上。

### 環境部署策略

**把 3090 遠端機器當成「研究實驗環境」，跟現有 docker 化的 SimLens 工程系統徹底分離**。原因：

- 現有的 docker SimLens 用 Replicate API + Gemini API，是**展示用 / 工程部署用**，不要動它
- 研究計畫的訓練 pipeline 應該獨立放在 3090 機器的 `~/simlens-research/` 目錄，產出 LoRA adapter + 報告數據
- 訓練完的 LoRA adapter 之後可以選擇是否反向整合到 docker SimLens 取代 Gemini

### 前 3 天執行順序

**Day 1：3090 環境體檢 + 工具鏈安裝**

bootstrap 腳本要做：
1. `nvidia-smi` 確認 driver / CUDA 版本
2. 確認 Python 版本(要 3.10+)
3. 安裝 conda/uv 環境管理
4. 建立兩個獨立 venv：
   - `simlens-perception`(Whisper + LLaVA-NeXT + transformers)
   - `simlens-train`(LLaMA-Factory + TRL + bitsandbytes + GPTQ)
5. 安裝 Ollama(給 Qwen3-32B-Q4 judge 用，最簡單)
6. 預先 pull：`whisper-large-v3`、`llama-3.2-3b-instruct`(要 HF token)、`qwen3:32b-q4_K_M`(Ollama)

**Day 2：Stage A pipeline 跑通(1 部影片)**

只做這件事 — 把 1 部 YouTube 影片，本地跑出 12 個 enriched_segment JSON：
```json
{"t_start": 0, "t_end": 10, "visual": "...LLaVA 描述...", "transcript": "...Whisper 文字..."}
```
跑通後，才確定 Stage A 整個 pipeline 在 3090 上是可行的。

**Day 3：Claude 蒸餾 sanity check(1 影片 × 1 persona × 12 段)**

只跑 12 個 Claude API call，看：
- prompt 格式對不對
- 輸出能不能正確解析(包含 None 反應)
- 一個 persona 的反應頻率分佈合不合理

跑通這 3 天，就有完整的「Stage A + Stage B 蒸餾資料生成」最小閉環，後面才規模化到 100 影片 × 8 persona。

### 省 API 錢的具體建議

研究計畫寫的 API 預算 **$120 USD 已經是最低底線**，能省的只有：

1. **Claude 蒸餾用 Haiku 不用 Sonnet** — 預算可以從 $115 砍到 ~$30。但研究計畫明確寫 teacher 用 Claude-3.5 Sonnet，理由是 student 要超越 teacher 才有 contribution。如果換 Haiku，論文「student 超越 600B teacher」的故事就弱很多。**不建議省這筆**。
2. **GPT-4o spot-check 用 GPT-4o-mini** — 從 $5 變 $0.5，影響不大，可省。
3. **如果 3090 上能跑 Llama-3.1-70B-Q4**(勉強可以，但很慢)，理論上可以當 teacher 取代 Claude，整套蒸餾改成全本地。但 70B 在 3090 上推 ~9600 次 prompt 會跑超久(估計 50+ 小時)，時間成本未必划算。

**結論**：API 預算就照計畫的 $120，這是研究嚴謹性的最低成本，省下去會傷論文。本地化的重點放在「訓練 + 推理 + judge」這三大塊（也是研究 contribution 的本體）。

### 為什麼用 WSL2 不用 Windows 原生 conda

研究計畫核心套件在 Windows 原生環境的支援度：

| 套件 | Windows 原生 |
|---|---|
| bitsandbytes（4-bit 量化） | 半官方，常踩坑 |

| LLaMA-Factory + 4-bit GPTQ + LoRA | 文件說支援，實測常出錯 |
| DeepSpeed | 不支援 |

**Why:** 研究計畫整套訓練 stack（LLaMA-Factory、TRL、PEFT、bitsandbytes、可能的 flash-attn）預設 Linux 環境。Windows 原生硬幹會花 30%+ 時間 debug 環境而不是做研究。

**結論**：3090 上 Windows 原生 Miniconda 留著沒關係（不衝突），但研究工作都在 WSL2 Ubuntu 內部做。

## 2026-04-28 — 訓練框架決策：保持 LLaMA-Factory

### 決策

維持原研究計畫的 **LLaMA-Factory** 作為 SFT + DPO 訓練框架，不改用純 HuggingFace（PEFT + TRL）或 Unsloth。

### 為什麼有這個討論

使用者問「為什麼不直接用 HuggingFace + Torch」。釐清後發現：LLaMA-Factory 本來就是建立在 HF + Torch 之上，依賴鏈為：

```
LLaMA-Factory → TRL + PEFT → Transformers → PyTorch
```

所以對立其實是「**抽象層級的選擇**」，不是「框架替換」。

### 比較三個層級的選項

| 方案 | 抽象 | 32 次訓練（8 LoRA × Phase 1+2 × 2 round）好不好做 | 6-aspect reward 客製化 | Skill 文件支援 |
|---|---|---|---|---|
| LLaMA-Factory（採用）| 高 | YAML config 跑迴圈 | 要繞框架 | 80 行（爬蟲）|
| Unsloth + TRL | 中 | Python script | 中等彈性 | 80 行（爬蟲）+ 1.8MB references |
| 純 PEFT + TRL | 低 | 寫 150-300 行 Python 迴圈 | 完全控制 | 855 行（手寫）|

### 觀察：Skill 完整度反映社群地位

- **PEFT / TRL** 是 HuggingFace 官方庫 → Skill 手寫 431 / 455 行
- **LLaMA-Factory / Unsloth** 是第三方 wrapper → Skill 自動爬文件 80 行

**完整度 ≠ 工具好不好**，但代表 debug 時我（Claude）能即時引用的教學內容多寡。

### 為什麼最終選 LLaMA-Factory

**Why:**
1. 學術界廣泛使用，論文寫法簡潔（"we use LLaMA-Factory"）
2. YAML config 可重現性高，比 Python script 易於分享
3. 4-bit GPTQ + LoRA + DPO 的組合已調好，避開常見踩坑
4. 32 次訓練用 32 個 YAML 比 32 段 Python 易維護
5. 使用者已熟悉，避免換工具的學習成本

**Trade-off 接受：**
- Phase 2 的 6-aspect reward 邏輯本來就在 LLaMA-Factory 之外做（Qwen judge 算分 → 寫 chosen/rejected JSONL → 餵給 LLaMA-Factory），這個分工 OK
- 出問題時要往 LLaMA-Factory → TRL → PEFT 三層挖，但實際 debug 機率不高
- Skill 文件量少 → debug 時靠搜 GitHub issues 補

### 何時應該重新考慮

- 如果 Phase 2 想做 **online RL**（生成 → 打分 → 立刻更新）→ LLaMA-Factory 不夠彈性，需切到 TRL 直接控制
- 如果 24GB VRAM 真的塞不下 → 切 Unsloth（2-5× 快、50-80% 省記憶體）
- 如果論文 reviewer 質疑「框架黑箱」→ 切純 PEFT + TRL

### 教授若問為什麼用 LLaMA-Factory

> 「LLaMA-Factory 是學術界 LLM 微調的標準框架，建立在 HuggingFace TRL + PEFT 之上。它提供 4-bit GPTQ + LoRA + DPO 的整合工作流，是 SimLens 32 次訓練 run 的最佳選擇 — 可重現性高、可維護性好。我們的 6-aspect reward 邏輯在框架外用 Python 實作，產出 chosen/rejected pairs 後餵給 LLaMA-Factory 做 DPO update。」

## 2026-05-04 — v4.1 評估體系大整修（多日決策一次性補登）

> 過去 6 天（4-28 → 5-04）紀錄缺漏，此筆把累積的評估體系決策、survey 設計、reward-eval 解耦修正一次性整理，方便之後追溯為什麼指標長這樣。

### Bilibili 路線完全砍除

之前 v4.0 想用 Bilibili 彈幕當「真人爆點 ground truth」驗證 R_timing。後來推敲下來：
1. Bilibili 用戶 vs YouTube 用戶 demographic 不同，爆點分布不可移植
2. 不同 persona 本來就有不同爆點，「整體爆點」這個概念與 SimLens per-persona 設計衝突
3. 找特定 persona 的彈幕觀看資料根本撈不到

**決定**：v4.1 完全砍 Bilibili，連 Group 5、Table 5、references [38][39] 全清乾淨。R_timing 改回純「Qwen judge 對單一 timestamp 的合理性評分」，不依賴外部 ground truth。

### Reward-Evaluation Contamination 修正

發現 v4.1 §5 評估體系 Tier 3 有兩個指標跟 Phase 2 reward 重疊：
- ❌ **Frequency Match Rate** ↔ R_frequency_match（兩者都量 list 長度匹配 persona 活躍度）
- ❌ **Coverage Spread** ↔ R_coverage_diversity（兩者都量 timestamp 分布均勻）

→ 跑出來的 evaluation 數字會「自我吹噓」（reward 直接優化的東西被當評估指標）。

**決定**：採方案 B，砍掉 Frequency Match Rate + Coverage Spread。Group 3 重新定位為「reward-independent metrics」。

### Persona Differentiation (Jaccard) 邏輯破口 → 換指標

原 Group 3 有 Jaccard timestamps 指標量「8 personas 的 timestamp 分布差異」，邏輯有破口：
- 共同爆點時 8 personas 自然會在同 timestamp 反應（影片 0:42 主持人摔手機，所有 persona 都會留言，只是 P1 心疼、P2 講螢幕、P5 說教、P8 玩梗）
- Jaccard timestamps 會把這個合理行為錯判為「LoRA 失敗」
- **應該量「反應內容」差異，不是「反應時間」差異**

**決定**：砍 Jaccard，換成 **Persona Content Distinctiveness (Embedding-based)** — OpenAI text-embedding-3-small 對每個 persona 的 concat comments 嵌入，8 個 persona 兩兩 cosine distance 取 28 對平均。

### Teacher Alignment Score (TAS) 補新指標

使用者問「蒸餾階段評估學生模型能在相同地方輸出評論的指標是哪個」 → 發現 v4.1 沒有量這個能力的指標，是個 gap。

**決定**：新增 **TAS (Teacher Alignment Score, F1@5s)** — Student 預測 timestamps vs Teacher (Claude) timestamps 的 temporal F1，借 SoccerNet (CVPR 2018) action spotting 的 ±tolerance window 範式。獨立於 reward（reward 只優化「絕對時點顯著性」R_timing，不直接優化「跟 Teacher 重合度」）。

### Empty-List Accuracy 砍除

使用者質疑 Empty-List Acc 在 SimLens 設計下沒用：
1. 跟 R_frequency_match 重複（list 長度涵蓋 len=0 特例）
2. Constrained decoding (Outlines/XGrammar) 已保證 schema 正確，「沒評論就不出 entry」是格式保證不是模型能力
3. Teacher 在 1-3min 短片上幾乎不會出空陣列，樣本太稀
4. TAS 的 F1 已涵蓋 false positive/negative 行為

**決定**：砍 Empty-List Acc。Group 3 從 3 個指標降為 2 個：TAS + Persona Content Distinctiveness。Table 3 從 3 欄縮成 2 欄。

### Survey 設計：對齊 SimTube IUI 2025 規格

原 v4.1 §5.2 Group 4 寫 25 人 Likert，但細節缺很多。對照 SimTube Section 6.1 的完整 protocol（WebFetch 確認）：
- 平台是 **Upwork** 不是 Prolific
- 25 人 × 8 影片 × 30 條評論/影片
- 統計用 **Wilcoxon Signed-Rank + Bonferroni**
- Quality control 5 階段：watch → quiz (≥80%) → summary → rate → feedback

**決定**：完全照搬 SimTube protocol，加 SimLens-specific 的 Timing Naturalness 維度 + 5 題 forced-choice A/B。Demographic-matched evaluators 不做（成本爆炸 8×、PersonaChat persona 太 niche 招不到、與 SimTube 失去可比性）。

> Side note：使用者後來表示「之後沒有要做這個我不做就好了」，但研究計畫上保留正式版 spec，方便未來投 journal 擴展時直接用。

### YouTube timed comments 重新評估

使用者點出「YouTube timed comments 還在啊」 → 確認 Type B（留言文字內含 `2:35` 自動變連結）一直存在。但對 SimLens 用處有限：
- 短片中 timed comments 比例 < 5%
- 抽出的留言都是「指向爆笑/精彩瞬間」，與 R_timing 訓練目標重合 → 不適合當 ground truth
- Persona 不可控

**決定**：不放進訓練/評估流程。Survey 對照組可選擇性加入但非必要。

### Table 1b Per-Persona Breakdown 聚合方式釐清

使用者問「Table 1b 的每個 cell 是把 persona 的多條評論平均嗎」→ 發現 v4.1 沒明寫聚合層級，不同維度算法其實不一樣：
- **per-list 級**：Persona Consistency, Coherence
- **per-persona-corpus 級**：Linguistic Habits, Distinct-N
- **per-comment 級**：Local Relevance (BERTScore), Engagingness (UniEval)

**決定**：補一張聚合方式表到 §5.2 Group 2 結尾，方便實作 reproduce 時對照。Std-Dev 行明確說明是「跨 8 persona 的 std」不是「persona 內部 std」。

### v4.1 Group 3 評估指標最終定版

```
Group 0: Schema Compliance Rate + Timestamp Validity Rate (deterministic)
Group 2: Tier 2 內容指標 (PersonaGym + Coherence + Engagingness + BERTScore + Distinct-N)
Group 3: List-level metrics (reward-independent)
         ├── TAS (Teacher Alignment, F1@5s)             ← 新增，借 SoccerNet
         └── Persona Content Distinctiveness (Embedding) ← 取代 Jaccard
Group 4: Human Eval (25 人 × Upwork × 8 影片 × 4 維度 × Wilcoxon+Bonferroni, 照 SimTube)
```

砍掉的指標清單（避免之後又把它們加回來）：
- ❌ Frequency Match Rate（跟 R_frequency_match 重疊）
- ❌ Coverage Spread（跟 R_coverage_diversity 重疊）
- ❌ Persona Differentiation Jaccard（共同爆點誤判為 LoRA 失敗）
- ❌ Empty-List Accuracy（被 R_frequency_match + TAS 涵蓋）
- ❌ Bilibili 爆點 recall（demographic 與 YouTube 不對齊）

## 2026-05-04 — 硬體升級：3090 24GB → RTX 5090 32GB

### 變動摘要

遠端訓練機器從 RTX 3090 24GB 升級到 **RTX 5090 32GB**（Ada Lovelace 後繼，5th-gen Tensor Cores）。

### 對 SimLens 訓練 pipeline 的具體影響

**1. VRAM 重新盤點（5090 32GB）**

| 模型 | 原 3090 占用率 | 5090 占用率 |
|------|--------------|------------|
| Whisper-Large-v3 (FP16) | 41% | 31% |
| LLaVA-NeXT-13B (4-bit) | 41% | 31% |
| Llama-3.2-3B (4-bit) + LoRA | 27% | 20% |
| Qwen3-32B (Q4_K_M) | 83%（吃滿） | 62% |

**2. 解鎖的關鍵能力：Llama-3B + Qwen judge 同卡共駐**

- 3090：必須 swap（生成 → 釋放 → 載 Qwen → 打分 → 釋放 → 載 Llama），每 round 切 4 次
- 5090：6.5 + 20 = 26.5GB / 32GB，**直接同卡並存**，Phase 2 RLAIF online judging 不用切換
- 估計整輪訓練 wall time **省 30-50%**

**3. 速度提升（Ada Lovelace vs Ampere）**

| 規格 | 3090 | 5090 | 倍數 |
|------|------|------|------|
| FP16 TFLOPS | 71 | ~165 | 2.3× |
| Memory Bandwidth | 936 GB/s | ~1792 GB/s | 1.9× |

**4. 預期時程壓縮**

| 階段 | 原估（3090） | 預期（5090） |
|------|------------|------------|
| Phase 1 SFT (8 LoRA × 100 樣本) | 4-6h | 2-3h |
| Phase 2 DPO (8 LoRA × 2 round) | 12-16h | 6-8h |
| Qwen judge (800 prompts × 4 candidates) | 6-8h | 3-4h |
| **Week 5-6 訓練+judging 總計** | **~30h** | **~15h** |

### 不改變的決策

**論文 framing 仍以「24GB 消費級 GPU 可重現」為基準**，因為：
1. SimLens 對比 SimTube/Claude 的核心賣點是 **可消費級部署**
2. 5090 32GB 只用於本研究訓練加速，不進入 paper claim
3. Reproducibility 章節列出最低配置 24GB + 4-bit GPTQ + LoRA 配置確保 24GB 可重現
4. 訓練超參全部按 24GB 上限設定，不利用 32GB 多出的餘裕（避免不可重現）

→ 所以 LoRA rank、batch size、gradient accumulation steps **全照 24GB 配置**，不調大。

### 不採用的選項（誘惑但有代價）

| 選項 | 為什麼不做 |
|------|-----------|
| 升級 Llama-3.2-3B → Llama-3.1-8B | 「3B 超越 600B Teacher」是論文核心賣點，換 8B 故事弱很多 |
| 跑全 BF16（不量化） | 跟「24GB 消費級部署」framing 衝突 |
| 並行 2 個 LoRA 訓練 job | 工程成本未必划算（要寫 multi-process 同步） |
| 利用多出 8GB 開更大 batch | 跟 24GB 重現性衝突，會讓論文配置不可移植 |

- ✅ **Flash Attention 2/3 全開**（5090 對 FA3 支援更好，省 20-30% memory + 加速）
- ✅ **Phase 2 同卡 online judging**（不切換模型，省 wall time）

### Prompt Caching 設計

每部影片 8 個 personas 共用同一個 Timeline Script (~2200 tokens)。將 prompt 拆成 3 個 content blocks：

```python
[
  {"type": "text", "text": persona_block},        # 不快取（每 persona 不同）
  {"type": "text", "text": timeline_block,
   "cache_control": {"type": "ephemeral", "ttl": "1h"}},  # ⭐ 快取
  {"type": "text", "text": tail_block},           # 不快取
]
```

效果：第 1 次 call 寫入 cache (1.25× input)，第 2-8 次讀取 cache (0.1× input)，Timeline Script 部分省 ~75%。配合 batch 50% 折扣再疊加 → 87.5% 總節省。

## 2026-05-06 — Anchor 設計大幅修正：砍 Bilibili + 抓 video-level data leakage

### 問題盤點（按發現時序）

**第 1 個問題：Bilibili 自動配對全錯**

`pair_yt_to_bili.py` 用 title-keyword 搜 Bilibili，22/49 能找到候選但**全部都是無關影片**（同名不同曲、relevant keyword 但完全不同內容）。實測例：
- YT "Pretty Little Baby - Connie Francis" → BV 找到「Lee Dungarees - Pretty Woman」（不同歌）
- YT "Chelsea 1-3 Nottingham Forest HIGHLIGHTS" → BV 找到「奥咕和秘密森林攻略」（遊戲攻略）

→ **跨平台配對需要人工判斷**（看縮圖 + 內容比對），程式做不到。

**第 2 個問題：YouTube heatmap 對 1-3 min 短片大部分 NA**

P0a sanity check 想用 YouTube most-replayed heatmap 當對照基準驗證 Bilibili，但 yt-dlp 測試顯示 1-3 min 短片普遍沒 heatmap。fallback 路線也斷。

**第 3 個問題（致命）：v4.2 §3.2 原設計有 video-level data leakage**

v4.2 §3.2 Step 1.4 寫「對每部影片（同 100 部 YouTube 影片）採 30 條留言當 Anchor A」+ 「同 50 部 Bilibili 配對當 Anchor B」 — **同 100 部影片同時是訓練資料 + 評估資料**。

```
v4.2 原設計：
  100 部 YouTube 影片
   ├── Stage A → Timeline Script → 蒸餾 → 訓練 SimLens 8 LoRA
   └── Anchor A / B 評估 ← 同 100 部抓留言/彈幕

→ SimLens 訓練看過影片 X 的 timeline，評估又問「對影片 X 的 hotspot 預測」
→ 等於用訓練集當測試集
```

v4.1 §3.3 雖寫 train/val/test = 0.85/0.10/0.05，但這是 **per (video, persona) pair 拆分**，每部影片仍會在 train+val+test 都出現（只是 8 personas 拆給不同 split），video-level 沒拆。

→ 這是 v4.2 設計層級的 bug，會讓 Anchor B per-video hotspot recall 數字虛高。

### 解決方案（採修法 B）

**修法 B = 額外撈 30 部 test set 影片，完全不進訓練**

```
新設計：
  100 部 (train)            ←── Stage A → Timeline Script → SimLens 訓練
  30 部 (test, 完全 hold out) ←── 所有評估（Anchor A 同部 sup. / Anchor B / Group 2 / 3 / 4）
```

**Anchor 設計同步修正：**

| Anchor | 之前（v4.2 原版）| 新設計（5/6 修正）|
|---|---|---|
| **A 文本錨** | 100 部 × 30 條真實留言（同訓練集，leakage）| HF `breadlicker45/youtube-comments-180k`（187K 條，獨立大池）+ 30 test 部對應留言（supplementary）|
| **B 時序錨** | Bilibili 50 部彈幕（跨平台配對失敗）| **30 test 部影片的 YouTube timestamp mentions**（per-video，同源 YouTube）|

**為什麼這個設計學術上更好：**
1. **完全消除 video-level data leakage**：30 test 從未進訓練
2. **per-video hotspot recall 仍能做**：同部影片才有 timestamp 對應意義
3. **Anchor B 訊號同源**：YouTube real users → YouTube real users（不需跨平台）
4. **無需人工配對**：YT 留言天然綁定影片
5. **無 IRB / 法律風險**：純 YouTube 公開資料
6. **Anchor A 大池更乾淨**：HF 180K 比 100×30=3000 條樣本量大 60×

### 學術論述：兩個 Anchor 各自的 claim

**Anchor A（distributional match, n=187K）**：
> SimLens 整體留言生態（length / sentiment / style / embedding 分布）比 Claude / GPT-4o-mini 更接近真實 YouTube 留言生態。
> 4 個 deterministic 指標、零 LLM judge、樣本量 187K → 統計顯著性最強。

**Anchor B（per-video hotspot recall @ ±5s, n=30 test videos）**：
> 在 30 部從未見過的測試影片上，SimLens 預測的 timestamp 對齊「真實 YouTube 觀眾在留言中主動標記的時點」recall=0.68 / precision=0.62，超越 Claude（0.51/0.48）+ GPT-4o-mini（0.46/0.43）。
> Per-video 嚴格對應、完全無 data leakage、樣本量 ~300 條 timestamp mentions。

兩個 anchor 互補：A 量「**怎麼寫**」、B 量「**何時寫**」、都跟 SimLens 任務直接相關。

## 2026-05-06 — v4.2.1 Anchor B 路線決策：採 B（distribution-level supplementary）

### 嘗試補強：撈 high-mention 候選影片

寫了 `collect_high_mention_videos.py`，用「mention-friendly」queries (compilations, top-N lists, reaction breakdowns, sports highlights, music mixes) 撈候選，每部 pre-fetch 50 條留言計 mention 數，min_mentions ≥ 5 才留。

**實測結果（YouTube API quota 耗盡前掃了 ~50 部）：**

```
Highlights/Compilation 類:
  KEEP 1GogvqHPy5E: 5 mentions in 50 comments
  KEEP cHQyo76Cgsw: 5 mentions in 50 comments
  skip: 0/50, 0/8, 2/50, 3/50, 2/39, 4/50, 0/50, 4/50, 0/50 ...
Reaction/Breakdown 類:
  KEEP N3LNdSDj5FI: 8 mentions in 48 comments
  KEEP 8KDuTVZgR0Y: 7 mentions in 50 comments
  skip: 1/50, 2/50, 0/50, 0/0, 0/0, 0/50 ...
（YouTube API quota 耗盡 → Sports/Music/Tutorial 類沒掃完）
```

掃了 ~50 部候選，**只有 4 部達標 mention ≥ 5**（8% 通過率）。

### 根本原因

1-3 min 短片觀眾普遍**不寫 timestamp mention**：
- 影片本身夠短，觀眾直接看完，不需要「跳到某時點」
- timestamp mention 集中在 **5+ 分鐘以上影片**：教學「跳到 5:30」、體育「7:42 神救援」、音樂 mix tracklist
- 即使 "top 10 moments" / "epic moments" / "compilation" 等 mention-friendly queries，1-3 min 短片仍少有 mention

→ 這是內容生態本質問題，**不能透過更好的 query 解決**。

### 決策：採選項 B（放棄 per-video Anchor B 強 claim）

兩個替代方案被否決：
- **選項 A（降 min-mentions 到 3 並補影片）**：明天 quota refresh 再撈，但實測 8% 通過率代表要掃 ~400 部才湊 30 部，且 mention=3 的 KDE peak 仍 noisy
- **選項 C（換 Anchor B 訊號源到 chapters/字幕）**：chapters 是創作者標的不是觀眾反應，違背 Anchor B「真實觀眾 reaction」精神

**選項 B 採用：**
- Anchor B 從「**per-video hotspot recall @ ±5s**」降為「**distribution-level temporal mention prior**」
- 78 條 mentions across 30 部 test 影片當「真實人類在 1-3 min 短片標記時點的整體 prior」
- 比 SimLens 整體 predicted timestamps 分布性質：
  - **Mention depth distribution**：mentions 落在影片前/中/後 1/3 比例
  - **Inter-mention gap distribution**：相鄰 mention 時間間隔分布
  - **Mention vs predicted distribution KS**：兩個分布的 KS 距離

## 2026-05-09 — v5.0 全新方向：PerVidCom + User-Archetype Multi-LoRA + Temporal CoT

### 大轉向動機

v4.x 路線（Bilibili / YouTube timestamp mentions / persona-driven hotspot detection）累積了一些工程資產（100 部影片 timeline scripts、8 個 P1-P8 persona），但研究故事一直在「真人爆點 ground truth 從哪來」這個問題上打轉，evaluation 缺乏乾淨的 baseline 對標。

今天讀完 PerVidCom 論文（Lin et al. EMNLP 2024）後決定全面轉向：把研究目標重新對齊到 **Personalized Video Comment Generation (PVCG)** 這個已有 benchmark、已有 baseline、已有 metric（FICL-Score）的任務。v5.0 是全新版本，**不繼承 v4.x 任何技術決策**。

### v5.0 核心架構

兩階段方法：

**Stage 1（碩論主體）— User-Archetype Multi-LoRA**
- PerVidCom 的 16,702 用戶用 style embedding（LUAR / Style-Embedding 候選）算 user-level 向量 → K-Means 分 K 個 archetype
- 每個 archetype 訓一個 LoRA（從 Video-LLaVA-7B-Instruct 開始）
- 推論時新用戶用少量歷史留言算 style vector → 路由到最近 archetype → 載入對應 LoRA 生成留言
- 對標 baseline：PV-LLM (Lin et al. 2024)、Gemini-1.5-Flash/Pro

**Stage 2（demo / 期刊延伸）— Temporal CoT**
- 純 prompt-based，**不重訓 LoRA**
- 模擬 agent 邊看影片邊產生內心 OS：t = 5/10/15/20s 各生一段 reasoning
- 最後基於所有 reasoning 收斂出 final comment
- UI 上配合進度條呈現「邊看邊想」效果（前端動畫 + 後台預先批次跑完 5 次 forward）

### 為什麼選 PerVidCom 不選 Personalized VideoIC

讀完 Wu et al. 2024 (WWW) 的 Personalized VideoIC：
- 來源是 Bilibili **彈幕**，不是一般留言 → 個人風格被彈幕文化稀釋
- 中文 + seen-user 切分 → 與 unseen-user 的研究主軸不符
- 雖然每用戶資料量較大（37.3 則 vs PerVidCom 的 ~21 則），但訊號品質弱

選 PerVidCom 的代價是 Shorts (~21 秒) 留言較短，但 unseen-user 設定 + 英文 + 一般留言（非彈幕）讓研究故事更乾淨。

### 為什麼最終 comment 選「reasoning-only」不選「融合 Stage 1 + reasoning」

考慮過三種 final comment 生成策略：
- 策略 A：純 reasoning → comment
- 策略 B：純 Stage 1 直接生成（reasoning 純展示）
- 策略 C：融合兩者

最終選 A。理由：
1. Thesis 乾淨（標準 CoT 範式）
2. Ablation 可比（Stage 1 vs Stage 2 直接證明 reasoning 貢獻）
3. 融合策略沒有理論依據，工程複雜度高
4. Stage 1 的「直接生成」仍保留作為 ablation baseline 行 + Demo Quick Mode

### Stage 1 必須做對讓 Stage 2 無痛接上的三件事

關鍵設計約束（為了讓 Stage 2 純 prompt 即可生 reasoning，不需重訓）：

1. **訓練 prompt 用自然語言指令式**，不用死格式 token
   - ✗ `[VIDEO][USER_TYPE_3]→COMMENT`
   - ✓ `"You are a viewer with style {desc}. Watch this video and write a comment..."`

2. **Base model 必須是 instruction-tuned 版本**
   - ✓ Video-LLaVA-7B-Instruct
   - ✗ 純 pretrained base

3. **LoRA 配置溫和，避免覆蓋掉 instruction-following**
   - rank=16, alpha=32, lr=5e-5, epochs=2

### Stage 1 驗收 Sanity Check（重要）

訓完 LoRA_k 後必須通過三個測試才視為完成：
1. `"Write a comment for this video as user type k"` → 應產生 comment
2. `"What would you think at the 5s mark?"` → 應產生 reasoning（不是 comment）
3. `"Output thoughts with [5s][10s][15s] tags then a final comment"` → 應產生結構化 trace

三個都過了，Stage 2 demo 用 prompt 即可實作；任一失敗則需降低 rank 或縮短 epochs 重訓。
